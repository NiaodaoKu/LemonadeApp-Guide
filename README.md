## 這個專案在做什麼

LexisPulse 是一個以間隔重複學習為核心的英語單字學習 App。它不是單純的單字本，而是把「輸入、複習、測驗、錯題回收、星號收藏、離線同步」串成完整流程，讓使用者可以每天持續使用，並在不穩定網路下仍保有學習進度。

這個專案解決的不是「能不能存單字」，而是「使用者怎麼持續學、怎麼快進快出、怎麼在弱網路或離線情境下不丟資料、怎麼讓每日複習真正形成習慣」。

---

## 產品特色

- 每日複習由系統自動安排，不需要使用者自己猜要複習哪些單字。
- 支援手動輸入、句子式輸入、批次貼上與掃描式匯入，降低資料建立成本。
- 提供多種測驗模式，讓同一組單字可以用不同方式練習。
- 星號單字與錯題庫可以把學習焦點拉回高價值內容。
- 離線可用，恢復網路後再補同步，避免弱網路時資料遺失。
- 多用戶資料彼此隔離，適合小範圍分享或個人使用。

---

## 核心功能

### 單字管理

- 手動輸入英文與中文對應資料。
- 支援一句一組、逐行輸入的句子模式。
- 新增時自動檢查重複，避免同一單字重複灌入。
- 可編輯、刪除、加星、標記為句子組。

### 複習系統

- 依輪次與距離上次複習的天數計算今日任務。
- 每次完成後更新輪次與下次複習節奏。
- 讓新字與老字都能有一致的回收機制。

### 測驗模式

- 英選中：看英文選中文。
- 中選英：看中文選英文。
- 聽力選意：透過發音判斷意思。
- 配對題：訓練中英文配對反應。
- 翻卡複習：適合快速掃過整批單字。

### 錯題與星號系統

- 錯題自動累積，連續答對後才逐步移出。
- 星號單字可做成高頻清單，方便集中複習。
- 這兩套機制讓使用者不只是做題，而是可以主動管理學習重點。

### 離線與同步

- 透過本地快取與佇列機制先保留使用者操作。
- 回到線上後再把變更補送到後端。
- 這個設計對日常通勤、訊號不穩或暫時斷線的使用情境特別重要。

## 架構總覽

```text
使用者介面 (Index.html)
  ├─ 單字輸入 / 編輯 / 測驗 / 錯題 / 星號
  ├─ 本地快取與離線佇列
  └─ 即時狀態更新
        |
        v
應用層 (Google Apps Script)
  ├─ API router
  ├─ 複習與測驗邏輯
  ├─ 資料驗證與重複檢查
  └─ 用戶隔離與狀態更新
        |
        v
資料層 (Google Sheets)
  ├─ 用戶索引
  ├─ 個人單字表
  ├─ 錯題庫
  └─ 星號與同步相關資料
```

### 為什麼這樣設計

- 前端直接承接使用者互動，讓操作反應快。
- 後端集中處理學習規則與資料邏輯，避免前端狀態太分散。
- Google Sheets 作為資料層，降低部署門檻與維護成本。
- 離線快取讓資料寫入不完全依賴即時網路。

## 技術亮點

### 1. 以學習流程為核心的資料設計

這個專案不是把單字資料平鋪而已，而是把它拆成可操作的學習狀態，例如輪次、剩餘複習次數、錯題連續正確數、星號收藏等。這讓系統可以從「字典」變成「學習引擎」。

### 2. 低成本但完整的全端結構

前端是單頁式 UI，後端用 Apps Script，資料存在 Google Sheets。這種做法的好處是維護成本低、部署速度快、適合個人專案與小型公開展示，也很適合作品集展示「我怎麼用有限資源做出完整產品」。

### 3. 離線優先與同步回補

許多學習工具在弱網路下體驗會直接掉線，但這個專案把離線視為正常使用情境。先在本地完成操作，再把變更同步到後端，可以顯著減少「做了一半結果丟失」的挫折感。

### 4. 多入口輸入能力

除了單筆新增，還支援批次貼上、句子模式與掃描式匯入。這對作品集很重要，因為它展現的不只是 UI，而是完整資料流設計：從內容建立到學習循環都被考慮進去。

### 5. 多種測驗模態

同一筆資料可以被不同題型重複使用，提升記憶強度，也讓測驗邏輯可以獨立於資料輸入流程。

---

## 資料結構概念

目前資料概念可以分成三層：

- 用戶層：保存使用者識別與基本狀態。
- 單字層：保存個別單字組、複習輪次、句子標記與輸入來源。
- 強化層：保存錯題、星號單字與其他學習焦點資料。

這種設計的重點是把「資料內容」和「學習狀態」拆開。單字本身是一份內容，而輪次、錯題次數、星號與今日剩餘次數是學習系統生成的狀態。這樣做的好處是後續要換資料庫時，資料邏輯比較容易遷移。

---

## 使用流程

1. 使用者登入後進入首頁。
2. 系統讀取用戶索引與個人資料，還原單字與學習狀態。
3. 使用者新增、編輯或匯入單字。
4. 每日進入時系統自動分配複習任務。
5. 透過測驗模式完成學習並累積輪次。
6. 錯題與星號資料持續反映使用者真正需要重點複習的內容。

這個流程代表專案的重點不是單一頁面，而是完整的學習閉環。

## 我做了哪些關鍵設計

- 建立了完整的單字學習流程，不只是資料 CRUD。
- 把複習輪次與錯題回收做成可持續運作的狀態機。
- 讓離線與弱網路情境不會直接破壞使用者體驗。
- 把重複檢查、輸入方式與測驗入口整合成同一套資料來源。
- 讓同一份資料可以同時服務「輸入」、「複習」、「測驗」與「收藏」。

如果要用作品集語言來說，這個專案展示的是：我不只會做畫面，我也會設計資料流、狀態管理與使用者行為循環。

## 與 Firebase 的關係

目前專案的展示版仍以現有資料層與後端設計為主，但未來會把後端資料庫搬到 Firebase。這部分在作品集中可以當成演進方向來寫，而不是主角。也就是說，README 的主線應該是 LexisPulse 本身做了什麼、怎麼運作、為什麼有價值；Firebase 則是下一階段的實作規劃。

簡單說，目前它已經是一個完整的學習產品展示案，Firebase 只是未來的資料承載選項。
